# Chapter 3: Historical Sessions and the ODT Challenge

> *The Kansas Legislature changed its website format multiple times between 2011 and 2026. Every era of the website requires its own parsing strategy — and the earliest era stores votes in a format you've probably never heard of.*

---

## Why Historical Data Matters

A single session is a snapshot. Fifteen years is a story.

With data from only the current legislature, you can measure where each legislator stands today. But you can't answer questions like: *Has the legislature become more polarized over time? Do legislators drift toward their party's mainstream the longer they serve? Did the Brownback tax experiment leave a lasting mark on Kansas politics?*

Those questions require historical data — the same measurements, applied consistently, across many sessions. Tallgrass covers the 84th through 91st Legislatures (2011–2026), eight bienniums spanning 15 years. With KanFocus data (Chapter 4), coverage extends back to 1999. But getting historical data is harder than getting current data, because the website evolved.

Imagine you're an archaeologist. The newest artifacts are on the surface, easy to reach and clearly labeled. But the deeper you dig, the more the tools and languages change. The artifacts are still there — you just need different techniques to read them.

## Three Eras of the Website

The Kansas Legislature's website has gone through three distinct eras in how it stores and presents legislative data:

### Era 1: The ODT Era (2011–2014, 84th–85th Legislatures)

In these sessions, individual vote records are not stored as web pages at all. They're stored as **ODT files** — OpenDocument Text format, the same format used by LibreOffice. When a legislator clicks a vote link, their browser downloads a document file instead of displaying a web page.

ODT files are actually ZIP archives containing XML inside. To extract vote data, the scraper has to:

1. Download the binary file
2. Unzip it
3. Find and read the XML inside
4. Parse the XML to extract vote metadata and individual legislator names

This is like arriving at the library and finding that the oldest volumes aren't books at all — they're sealed envelopes containing handwritten notes. You need completely different tools to read them.

### Era 2: The JavaScript Era (2015–2020, 86th–88th Legislatures)

By the 86th Legislature (2015–2016), vote records switched to regular HTML pages — the same format used today. But the bill listing pages were still generated by JavaScript. This means the scraper can't find bills by simply reading the HTML of the listing page; it has to download the JavaScript data files and parse them.

### Era 3: The Modern Era (2021–2026, 89th–91st Legislatures)

From the 89th Legislature onward, everything is plain HTML. Bill listings, vote pages, and legislator profiles are all standard web pages that the scraper can read directly. This is the simplest era to scrape.

Here's the full compatibility matrix:

| Session | Years | Bill Discovery | Vote Format | Party Detection |
|---------|-------|---------------|-------------|-----------------|
| 91st | 2025–2026 | HTML pages | HTML pages | `District N - Party` |
| 90th | 2023–2024 | HTML pages | HTML pages | `District N - Party` |
| 89th | 2021–2022 | HTML pages | HTML pages | `District N - Party` |
| 88th | 2019–2020 | JavaScript (quoted keys) | HTML pages | `District N - Party` |
| 87th | 2017–2018 | JavaScript (unquoted keys) | HTML pages | `District N - Party` |
| 86th | 2015–2016 | JavaScript (unquoted keys) | HTML pages | `Party: Name` |
| 85th | 2013–2014 | JavaScript (unquoted keys) | ODT files | `Party: Name` |
| 84th | 2011–2012 | JavaScript (unquoted keys) | ODT files | `Party: Name` |

Notice that the transitions don't all happen at the same time. Vote format changed between the 85th and 86th, bill discovery changed between the 88th and 89th, and party detection changed between the 86th and 87th. The scraper has to handle every combination correctly.

**Codebase:** `src/tallgrass/session.py` (`uses_odt` property, `js_data_paths`, URL pattern logic)

## Parsing ODT Vote Files

Let's walk through how the ODT parser works, since it's the most unusual component of the scraper.

### What's Inside an ODT File

An ODT file is a ZIP archive. When you unzip it, you find several XML files. The one that matters for votes is called `content.xml`. Inside that file, the vote data is stored in two places:

**1. Metadata fields** — structured data about the vote:
- Chamber (House or Senate)
- Bill number
- Date and time (e.g., "2011/02/25 11:43:27")
- Vote tally (e.g., "66:53")
- Action code (e.g., "fa_fabc_341" for "Final Action - Failed as Amended by Committee")

**2. Body text** — the actual list of legislators and how they voted, formatted as human-readable paragraphs:
```
Yeas (66):
  Alford, Arpke, Aurand, Bethell, ...

Nays (53):
  Ballard, Burroughs, Carlin, ...

Absent and Not Voting (6):
  Grant, O'Neal, Powell, ...
```

The parser extracts the metadata from XML elements and the vote lists from the body text using pattern matching.

### The Name Resolution Problem

HTML vote pages identify legislators by a URL slug — a machine-friendly identifier like `sen_smith_john_1`. This slug is unique and consistent across an entire session.

ODT files don't have slugs. They have names — just names. "Smith" or "C. Holmes" or "O'Neal." And names are ambiguous. Is "Holmes" Representative Charles Holmes or Representative Dennis Holmes? Is "O'Neal" the same person who appears as "O'Neal" in one file and "ONeal" in another?

The parser solves this with a **member directory** — a lookup table built from the legislature's member listing page. For each legislator, the directory maps their last name (or initial-qualified name) to their unique slug.

The resolution process:

1. **Exact match:** Look up the full name in the directory. If it matches exactly one person, done.
2. **Initial-qualified match:** If the ODT says "C. Holmes," look for a legislator whose first name starts with "C" and last name is "Holmes."
3. **Ambiguity flag:** If two legislators share a last name and no initial is provided, the match is flagged as ambiguous. These cases are logged but not guessed — better to record "unknown" than to assign a vote to the wrong person.
4. **New legislator:** If no match exists at all (perhaps a special election brought in someone who wasn't in the original directory), the parser creates a new record.

### The 30% Problem

Here's an edge case that caused confusion early on: in the 84th Legislature (2011–2012), roughly 30% of ODT files contain only tally metadata — the total count of Yea and Nay votes — but no individual legislator names. These are committee-of-the-whole votes, where the full chamber debates a bill as if it were a committee. The resulting ODT records the outcome (66-53) but not who was on which side.

This is not a parser bug. The data genuinely isn't there. The parser handles this gracefully: it extracts whatever metadata exists and returns empty vote lists for these files. Downstream phases simply have fewer data points for the 84th session.

**Codebase:** `src/tallgrass/odt_parser.py` (all parsing functions), `src/tallgrass/scraper.py` (`_parse_odt_vote_pages()` and member directory construction)

## Parsing JavaScript Bill Data

For sessions before 2021, the bill listing pages are generated by JavaScript. When the scraper downloads these pages, it gets something like:

```html
<html>
  <script src="/bills_li_2018.js"></script>
  <div id="bill-list"></div>
</html>
```

The actual bill data is in the JavaScript file, not the HTML. The scraper downloads the JavaScript file directly and parses it.

### The Two Key Formats

The JavaScript data files come in two flavors:

**Quoted keys (88th Legislature, 2019–2020):**
```javascript
var measures_data = [
  {"title": "SB 1", "measures_url": "/li_2020/b2019_20/measures/sb1/"},
  {"title": "SB 2", "measures_url": "/li_2020/b2019_20/measures/sb2/"},
  ...
];
```

This is valid JSON, so the parser can read it directly.

**Unquoted keys (84th–87th Legislatures, 2011–2018):**
```javascript
var measures_data = [
  {title: "SB 1", measures_url: "/li_2018/b2017_18/measures/sb1/"},
  {title: "SB 2", measures_url: "/li_2018/b2017_18/measures/sb2/"},
  ...
];
```

This is valid JavaScript but *not* valid JSON. The difference is subtle — `title` vs. `"title"` — but a JSON parser will choke on the first format. The scraper's `parse_js_array()` function detects unquoted keys and adds the missing quotes before parsing.

**Think of it this way.** Imagine receiving a spreadsheet where some cells are properly formatted and others are informal shorthand. "Name: John" is clear enough, but a computer program that expects strict formatting will reject it. The parser acts as a translator, normalizing the informal JavaScript into the strict JSON that Python can understand.

**Codebase:** `src/tallgrass/bills.py` (`parse_js_array()`, `parse_js_bill_data()`)

## Session URL Archaeology

The Kansas Legislature's website uses different URL patterns for different sessions, and the mapping isn't always obvious:

```
Current session (2025-2026):
  /li/b2025_26/measures/bills/

One session back (2023-2024):
  /li_2024/b2023_24/measures/bills/

Special session (2024):
  /li_2024s/b2023_24/measures/bills/
```

The pattern is: the current session uses `/li/`, historical sessions use `/li_{end_year}/`, and special sessions use `/li_{end_year}s/`. But what counts as "current" changes when a new biennium begins — and there's no automated detection. The scraper has a constant (`CURRENT_BIENNIUM_START`) that must be updated manually when a new session starts.

Special sessions are even more idiosyncratic. Their URL patterns don't follow a consistent formula — each special session's paths were determined empirically by inspecting the actual URLs on the live site. The scraper stores these as a lookup table.

The KLISS API has its own versioning: current sessions use `/li/api/v13/rev-1`, while historical sessions use `/li_{end_year}/api/v13/rev-1`. The API version number (v13/rev-1) has remained stable across recent sessions, but earlier sessions used earlier versions.

**Codebase:** `src/tallgrass/session.py` (`KSSession` class — URL construction properties)

## Special Sessions: The Merge Problem

Kansas occasionally convenes special sessions — emergency legislative sessions called by the Governor outside the regular schedule. Five special sessions appear in the Tallgrass data: 2013, 2016, 2020, 2021, and 2024.

Special sessions produce their own vote data, but they logically belong to their parent biennium. The 2024 special session, for example, was part of the 90th Legislature (2023–2024). For the analysis pipeline to work correctly, special session votes need to be merged into the parent session's data files.

The merge process is straightforward but carefully designed to be **idempotent** — meaning you can run it multiple times and always get the same result:

1. Read the parent session's CSV files
2. Remove any previously merged special session rows (identified by their session label, e.g., "2024 Special")
3. Read the special session's CSV files
4. Concatenate the two datasets
5. Deduplicate legislators (the parent session's record wins if there's a conflict)
6. Write the combined result back to the parent's files

The idempotent design means you never have to worry about "did I already merge this?" Running the merge a second time simply removes the old merge and re-applies it, producing identical results.

```bash
# Merge the 2024 special session into the 90th Legislature
just scrape --merge-special 2024

# Merge all known special sessions
just scrape --merge-special all
```

**Codebase:** `src/tallgrass/merge_special.py` (`merge_special_into_parent()`)

---

## Key Takeaway

The Kansas Legislature's website has changed formats three times since 2011: ODT vote files became HTML, JavaScript bill lists became HTML, and party detection patterns shifted. The scraper handles all three eras behind a unified interface, routing each session to the correct parser based on its year. The result is 15 years of consistently structured data, ready for analysis, regardless of how the source website evolved.
