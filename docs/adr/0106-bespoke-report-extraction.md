# ADR-0106: Bespoke Multi-Report Section Extraction

**Date:** 2026-03-10
**Status:** Accepted

## Context

The 27-phase pipeline produces comprehensive HTML reports (10–25 sections each, 1–5 MB per report). These are the right output for analysis but the wrong output for presentations, stakeholder briefings, or answering a specific research question.

Sharing pipeline results with non-technical audiences (journalists, policymakers) requires either sending entire multi-megabyte reports or manually copy-pasting HTML — tedious, error-prone, and not reproducible. A presentation might need an EDA overview from Phase 01, IRT ideal points from Phase 05, and a network visualization from Phase 11 — sections from three different reports — combined into a single shareable file.

## Decision

Implement `tallgrass-extract`, a CLI tool that extracts sections from one or more pipeline HTML reports into a standalone, self-contained HTML file.

### Key design choices:

1. **Parse, don't re-render.** The HTML is machine-generated by `analysis/report.py`'s Jinja2 template with a predictable `<section>` structure. Regex extraction is fast, reliable, and requires no data loading or model re-runs.

2. **Multi-report composition.** Each report path on the CLI starts a group; `--section` flags bind to the most recent report. Sections from different reports carry provenance annotations showing their source phase and session.

3. **CSS extracted from source HTML.** The tool doesn't import from the `analysis` package (which isn't on `sys.path` for entry-point scripts). Instead, it extracts the `<style>` block from the first source report, with a minimal fallback for edge cases.

4. **CDN dependency deduplication.** Plotly and DataTables scripts embedded per-section are collected, removed from section bodies, and injected once into `<head>`.

5. **Separate output directory.** Extractions go to `results/bespoke/<date>/` with a `latest` symlink, never touching the pipeline's `results/kansas/` tree.

### Architecture:

```
src/tallgrass/extract/
  __init__.py      - Package init
  extractor.py     - Core: list_sections(), extract_sections(), render_extracted()
  cli.py           - CLI entry point with multi-report argument parsing
```

### CLI:

```bash
# List sections
just extract --list report.html

# Single report
just extract report.html --section 15 --title "My Extract"

# Multi-report composition
just extract \
    report_a.html --section 15 \
    report_b.html --section 3 --section 5 \
    --title "Combined Presentation"
```

### Section matching:

- By number: `--section 15` matches `<span class="section-number">15.</span>`
- By ID: `--section interactive-2d-scatter-senate` matches `id="..."`
- Detection is automatic: pure integer → number, otherwise → ID

## Consequences

**Benefits:**
- Reproducible extraction — same command produces same output
- Multi-report composition enables presentation-quality deliverables from pipeline output
- No re-computation — operates on finished HTML in milliseconds
- Preserves all interactivity (Plotly hover, DataTables search/sort)
- Original section numbering maintained for traceability

**Trade-offs:**
- Regex-based parsing assumes the exact HTML structure from `report.py` — a template change would require updating `SECTION_RE`
- CSS comes from source reports, not a canonical constant — theoretically different reports could have different CSS versions (unlikely in practice since they're generated by the same code)

**Not in scope:**
- PDF export (browser print-to-PDF works for static sections)
- Re-rendering from source data (if the report doesn't exist, run the pipeline first)

**Testing:** 50 tests covering parsing, extraction, dependency dedup, rendering, CSS handling, and CLI.
