# Bespoke Report Extraction

**Date:** 2026-03-10
**Status:** Implemented

## Problem

The 27-phase pipeline produces comprehensive HTML reports — 10–25 sections each, covering convergence diagnostics, full legislator tables, correlation plots, and interactive visualizations. These are the right output for analysis. They are the wrong output for presentations, stakeholder briefings, or answering a specific research question.

When a journalist asks "show me where Huelskamp sits relative to the rest of the 79th Senate," the answer is section 15 of the 06_IRT_2D report. But sharing that means either sending the entire 2MB report or manually extracting HTML — tedious, error-prone, and not reproducible.

A presentation might need sections from several different phase reports — an EDA overview, IRT ideal points, and a network visualization — combined into a single shareable file.

## Solution

A CLI tool (`tallgrass-extract`) that extracts sections from one or more pipeline HTML reports into a standalone, self-contained HTML file. The tool operates on finished HTML (no re-computation), outputs to a separate `results/bespoke/` directory, and preserves all interactivity (Plotly hover, DataTables sorting).

Multi-report extraction is the primary use case: cherry-pick sections from different phase reports and compose them into a single presentation-ready page.

## Design Principles

1. **Parse, don't re-render.** The HTML reports are machine-generated with predictable structure. Extracting sections from the finished HTML is fast, reliable, and doesn't require loading data or re-running models.

2. **Separate output directory.** Bespoke extractions go to `results/bespoke/<date>/`, never touching the pipeline's `results/kansas/` tree. A `latest` symlink follows the existing convention.

3. **Sections are the unit of extraction.** Every report section has a unique `id` attribute and a sequential number. Both are valid selectors.

4. **Preserve interactivity.** Plotly charts, ITables, and DataTables all work in the extracted output. CDN script tags are deduplicated and hoisted into `<head>`.

5. **Multi-report composition.** Each report path starts a group; `--section` flags bind to the most recent report. Sections from different reports carry provenance annotations.

## CLI Interface

```bash
# List all sections in a report
just extract --list results/kansas/79th_2001-2002/latest/06_irt_2d/06_irt_2d_report.html

# Extract by section number (single report)
just extract results/kansas/79th_2001-2002/latest/06_irt_2d/06_irt_2d_report.html \
    --section 15

# Extract by section ID
just extract results/kansas/79th_2001-2002/latest/06_irt_2d/06_irt_2d_report.html \
    --section interactive-2d-scatter-senate

# Extract from multiple reports into one file
just extract \
    results/kansas/91st_2025-2026/latest/06_irt_2d/06_irt_2d_report.html --section 15 \
    results/kansas/91st_2025-2026/latest/01_eda/01_eda_report.html --section 3 --section 5 \
    results/kansas/91st_2025-2026/latest/24_synthesis/24_synthesis_report.html --section maverick-profiles-house \
    --title "91st Legislature: Key Findings" \
    --output 91st-presentation.html

# Custom title and output path
just extract results/kansas/79th_2001-2002/latest/06_irt_2d/06_irt_2d_report.html \
    --section 15 \
    --title "79th Senate: 2D Ideal Points" \
    --output senate-huelskamp.html
```

### Argument Grammar

A new report group starts whenever a positional argument (report path) is encountered. `--section` flags bind to the most recent report. Global flags (`--title`, `--output`, `--list`) apply to the whole extraction.

### Arguments

| Argument | Required | Description |
|----------|----------|-------------|
| `report` | Yes | Path(s) to source HTML report(s) — positional, repeatable |
| `--section` | No* | Section ID or number (repeatable). Binds to preceding report. Required unless `--list`. |
| `--list` | No | Print all sections (number, ID, title) for the first report and exit |
| `--title` | No | Custom title for extracted report (default: "Extracted Report") |
| `--output` | No | Output filename under `results/bespoke/<date>/` (default: slug from title) |

## HTML Parsing Strategy

Pipeline reports have a predictable structure generated by `ReportBuilder` in `analysis/report.py`:

```html
<section class="report-section" id="SECTION_ID">
    <h2><span class="section-number">N.</span> Section Title</h2>
    ... content (tables, figures, Plotly divs, text) ...
</section>
```

A regex captures all four components (ID, number, title, content) in a single pass:

```python
SECTION_RE = re.compile(
    r'<section class="report-section" id="([^"]+)">\s*'
    r'<h2><span class="section-number">(\d+)\.</span>\s*(.*?)</h2>'
    r'(.*?)</section>',
    re.DOTALL,
)
```

This is safe because the HTML is machine-generated by our own template — not arbitrary user HTML.

### Section matching

The `--section` argument supports two modes:

1. **By number:** `--section 15` matches `<span class="section-number">15.</span>`
2. **By ID:** `--section interactive-2d-scatter-senate` matches `id="interactive-2d-scatter-senate"`

Detection is automatic: if the argument is a pure integer, match by number; otherwise match by ID.

## Dependency Handling

### Plotly

Each `InteractiveSection` includes its own Plotly CDN `<script>` tag. When extracting multiple interactive sections (especially across reports), these would be duplicated. The extractor:

1. Scans all extracted content for Plotly CDN script tags
2. Removes duplicates from section bodies
3. Injects a single copy into `<head>`

### ITables / DataTables

Same pattern — interactive tables embed their own JS/CSS inline. The extractor deduplicates library loads while preserving per-table initialization scripts.

### Base64 images

Already self-contained in `<img src="data:image/png;base64,...">`. No special handling needed.

### CSS

Reuses `REPORT_CSS` from `analysis/report.py` directly — same styling as the full report.

## Output Structure

```
results/
  bespoke/
    2026-03-10/
      91st-presentation.html
      79th-senate-2d-ideal-points.html
    latest -> 2026-03-10/
```

Default slug generation from `--title`. With `--output`, the file is written to the specified filename under `results/bespoke/<date>/`.

## Provenance

When extracting from multiple source reports, each section gets a provenance annotation:

```html
<p class="provenance">From 06_irt_2d — 91st_2025-2026</p>
```

For single-source extractions, provenance is omitted (redundant).

## Rendered Output

The extracted HTML is a minimal, self-contained page:

- **Header:** Custom title (or "Extracted Report"), source count, extraction timestamp
- **TOC:** Only if multiple sections extracted (with source provenance when multi-report)
- **Sections:** Extracted content with original numbering preserved
- **Footer:** Generation timestamp
- **Styling:** Same `REPORT_CSS` as pipeline reports — visually identical

## Example: Multi-Report Presentation

```bash
just extract \
    results/kansas/79th_2001-2002/latest/01_eda/01_eda_report.html --section 1 \
    results/kansas/79th_2001-2002/latest/06_irt_2d/06_irt_2d_report.html \
        --section interactive-2d-scatter-senate \
    results/kansas/79th_2001-2002/latest/24_synthesis/24_synthesis_report.html \
        --section maverick-profiles-senate \
    --title "79th Kansas Senate: Three Factions"
```

This produces a single HTML file combining the EDA overview, the interactive 2D IRT scatter, and the maverick profiles — each with provenance showing its source phase.

## Not In Scope

- **Re-rendering from source data.** This tool extracts from finished HTML. If the source report doesn't exist, run the pipeline first.
- **PDF export.** HTML is the output format. Browser print-to-PDF works for static sections; Plotly charts require a screenshot.
